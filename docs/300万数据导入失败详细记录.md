## 五、错误诊断

### 5.1 查看 Doris 错误日志
```bash
curl -s "http://127.0.0.1:8040/api/_load_error_log?file=..."
```

### 5.2 错误详情
```
Reason: column(emp_id) values is null while columns is not nullable. 
src line [NULL NULL NULL NULL NULL NULL NULL 1]; 

Reason: column(emp_id) values is null while columns is not nullable. 
src line [NULL NULL NULL NULL NULL NULL NULL 1]; 

... (重复错误)
```

### 5.3 问题分析

#### 问题 1：列名不匹配
- **Doris 期望的列名**：`emp_id`
- **实际数据的列名**：`id, name, amount, status, created_at, description`
- **原因**：Doris 表结构定义与实际数据不匹配

#### 问题 2：数据格式问题
从错误日志看，Doris 接收到的数据格式为：
```
[NULL NULL NULL NULL NULL NULL NULL 1]
```
这表明：
1. JSON 数据没有被正确解析
2. 所有字段都被解析为 NULL
3. 只有最后一个字段被识别为 `1`

#### 问题 3：可能的根本原因
1. **JSON 格式问题**：Jackson 序列化时，`created_at` 字段（TIMESTAMP 类型）可能被序列化为数字而不是字符串
2. **列映射问题**：Doris 无法将 JSON 字段映射到表列
3. **数据类型不匹配**：MySQL 的 TIMESTAMP 与 Doris 的 DATETIME 格式不兼容

## 七、解决方案

### 7.1 方案 1：修复 Doris 表结构
**问题**：Doris 表期望 `emp_id` 列，但数据中没有这个字段。

**解决**：重新创建 Doris 表，确保列名与数据匹配。

```sql
-- 删除旧表
DROP TABLE IF EXISTS test_db.test_target_table;

-- 创建新表（与数据列名一致）
CREATE TABLE test_target_table (
    id BIGINT,
    name VARCHAR(100),
    amount DECIMAL(10, 2),
    status INT,
    created_at DATETIME,
    description VARCHAR(500)
) ENGINE=OLAP
DUPLICATE KEY(id)
DISTRIBUTED BY HASH(id) BUCKETS 10
PROPERTIES ("replication_num" = "1");
```

### 7.2 方案 2：修复数据格式
**问题**：`created_at` 字段可能被序列化为数字。

**解决**：在 DataPipeline 中显式转换日期格式。

```java
// 在 DataPipeline.produce() 方法中
for (int i = 1; i <= columnCount; i++) {
    Object value = rs.getObject(i);
    String columnName = metaData.getColumnLabel(i);
    
    // 特殊处理日期类型
    if (value instanceof java.sql.Timestamp) {
        value = new SimpleDateFormat("yyyy-MM-dd HH:mm:ss")
                .format((java.sql.Timestamp) value);
    }
    
    row.put(columnName, value);
}
```

### 7.3 方案 3：添加列映射
**问题**：Doris 无法自动映射 JSON 字段到表列。

**解决**：在 Stream Load 请求中添加列映射。

```java
// 在 StreamLoadService.doLoad() 方法中
conn.setRequestProperty("columns", "id, name, amount, status, created_at, description");
```

## 九、经验总结

### 9.1 关键教训
1. **小规模测试通过不代表大规模测试通过**：8,988 条成功不代表 300 万条也能成功
2. **数据类型一致性**：MySQL 和 Doris 的数据类型需要严格对应
3. **日期格式处理**：TIMESTAMP 类型需要显式转换为字符串格式
4. **列名映射**：确保 JSON 字段名与 Doris 表列名完全一致

### 9.2 最佳实践
1. **分阶段测试**：先测试小规模（1 万条），再测试中规模（10 万条），最后测试大规模（300 万条）
2. **详细日志**：记录每个批次的导入详情，便于问题定位
3. **错误日志分析**：及时查看 Doris 的错误日志，快速定位问题
4. **数据验证**：导入后立即验证，确保数据完整性

### 9.3 性能优化建议
1. **批次大小**：50,000 条/批是合理的，平衡内存和效率
2. **压缩**：当前 Docker 环境不支持 gzip，暂关闭
3. **重试机制**：3 次重试 + 指数退避是合理的
4. **断点续传**：大数据量下必须支持断点续传
