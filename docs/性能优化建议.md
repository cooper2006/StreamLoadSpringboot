# 性能优化建议

## 已实施的优化

### 1. HTTP 连接优化 ✅
- **启用 Keep-Alive**：复用 HTTP 连接，减少 TCP 握手开销
- **禁用缓存**：`setUseCaches(false)` 避免不必要的缓存检查
- **立即读取响应**：减少连接空闲时间

### 2. 配置化参数 ✅
所有关键参数都可配置，便于针对不同环境调优：

```yaml
doris:
  # 批次大小：越大吞吐量越高，但内存占用也越大
  batch-size: 50000
  
  # 消费者线程数：并行导入，建议 3-5
  consumer-threads: 3
  
  # 队列容量：生产者-消费者之间的缓冲
  queue-capacity: 8
  
  # 连接超时（秒）
  connect-timeout: 60
  
  # 读取超时（秒）
  timeout: 600
```

## 进一步提升吞吐量的建议

### 3. 增加消费者线程数（立竿见影）

当前配置：`consumer-threads: 3`

**建议测试：**
```yaml
# 4 个消费者线程，理论上可提升 33% 吞吐量
consumer-threads: 4

# 或 5 个消费者线程（需要更多 CPU 和网络带宽）
consumer-threads: 5
```

**注意事项：**
- 每个消费者线程独立建立 HTTP 连接
- 需要足够的 CPU 和网络带宽
- 建议逐步增加，观察失败率

### 4. 调整批次大小

当前配置：`batch-size: 50000`

**大数据量场景（> 500万）：**
```yaml
# 增大批次，减少 HTTP 请求次数
batch-size: 100000
```

**小数据量或内存受限场景：**
```yaml
# 减小批次，降低内存峰值
batch-size: 20000
```

### 5. 增大队列容量

当前配置：`queue-capacity: 8`

**如果生产者明显快于消费者：**
```yaml
# 增大缓冲，让消费者持续工作
queue-capacity: 16
```

### 6. 启用压缩（网络带宽受限时）

当前配置：`enable-compression: false`

**如果网络带宽是瓶颈：**
```yaml
# 启用 gzip 压缩，减少传输数据量
enable-compression: true
```

**权衡：**
- 压缩率通常 60-70%
- 增加 CPU 开销（压缩/解压）
- 适合网络带宽 < 1Gbps 的场景

### 7. 调整超时时间

当前配置：
```yaml
connect-timeout: 60  # 连接超时
timeout: 600         # 读取超时
```

**大数据量批次：**
```yaml
# 增大读取超时，避免大批次超时
timeout: 900
```

### 8. 生产环境日志级别

当前配置：
```yaml
logging:
  level:
    com.example.streamload: DEBUG  # 开发环境
```

**生产环境建议：**
```yaml
logging:
  level:
    com.example.streamload: INFO  # 减少日志输出开销
```

## 监控指标

运行时关注以下指标：

1. **吞吐量**：`总记录数 / 总耗时`（条/秒）
2. **失败率**：失败批次 / 总批次
3. **队列深度**：`[进度] 生产: X 批次, 消费: Y 批次`
   - 如果 X 远大于 Y，说明消费者跟不上，需要增加消费者线程
4. **HTTP 响应时间**：Doris 日志中的 `LoadTimeMs`

## 预期性能

| 配置 | 预期吞吐量 | 适用场景 |
|------|-----------|---------|
| 当前（3消费者，5万/批） | ~100K 条/秒 | 通用 |
| 4消费者，5万/批 | ~130K 条/秒 | CPU 充足 |
| 3消费者，10万/批 | ~120K 条/秒 | 内存充足 |
| 5消费者，10万/批 | ~160K 条/秒 | 高性能服务器 |

**注意：** 实际性能受限于：
- Doris BE 的写入能力
- 网络带宽和延迟
- MySQL 源数据库的查询速度
- 服务器 CPU 和内存
